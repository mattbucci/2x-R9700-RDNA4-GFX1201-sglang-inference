# 256K perf resweep on v0.5.13.post1 (2026-06-16)

Re-measuring single-user 256K decode perf on the **promoted v0.5.13.post1 stack** (env
`sglang-triton36-v0513`, tree `/data/sgl-rebase`) vs the v0.5.12 baselines in README, after the
2026-06-16 rebase + promotion. Each model: boot via the default `launch.sh` preset, measure with
`scripts/bench/bench_all_unified.py` (context-depth TPOT→tok/s curve) + a direct server-log
gen-throughput point at true ~230K depth for the @256K headline. Capturing every v0.5.13 serving
issue as a patch (per the rebase discipline).

## North-Mini-Code-1.0-fp8 (`cohere2_moe`, MoE-128e + hybrid-SWA, FP8) — ✅ parity, 1 patch

**Serving: required patch 062** (cohere2moe-v0513-hybrid-swa-classification). v0.5.13 has **native**
cohere2_moe (model + config + `cohere_command4` reasoning/tool parsers) → the v0.5.12 CANDIDATEs
051/052/053/054 are mostly **subsumed**. The surviving delta: v0.5.13 doesn't classify
`Cohere2MoeForCausalLM` as hybrid-SWA, so the triton SWA decode crashed at cuda-graph capture
(`window_kv_indptr=None` → `AttributeError: 'NoneType'.shape`), then `get_hybrid_layer_ids` returned
None. Fix = allowlist entry + `layer_types` branch (see `patches/062-*`). v0.5.13's native RoPE/dense
handling is correct (config-pop NoPE bug **not** present — the v0.5.12 052 force_rope fix is subsumed).

**Capability (32K boot):** L0-RoPE positional gate **PASS** (`ember,copper`); thinking composes
(reasoning_content + clean content); **no `special=False` delimiter leak**. SWARadixCache
`hybrid_swa=True`, 1.58M-tok KV pool at 262144.

**Decode tok/s (cuda-graph ON, single-bs, single-user):**

| depth (input) | v0.5.13 | v0.5.12 baseline | note |
|---|---|---|---|
| short (≤1K)   | **65.4** | 65.4 | **exact match** |

**Verdict: v0.5.13 holds North-Mini perf** — short-context is an exact match (65.4) and the whole
curve tracks the v0.5.12 baseline; the deepest point (~31 @231K vs 33.6 @256K) is within
measurement+depth differences (no material regression). Method notes: `bench_all_unified` uses
input=ctx/2 so its deepest is 131K-input (41.3, labeled ctx=262144); the true ~231K-depth number is
the direct server-log gen-throughput (`scripts/.../deep_decode.py`), content-independent for plain

## Nemotron-3-Nano-Omni-30B-A3B FP8 (`nemotron_h` Mamba2+MoE hybrid AVLM) — ✅ parity-or-better, 1 patch + 1 env

Needed **two fixes** to serve on v0.5.13:
1. **env: `librosa` missing** (now in `setup.sh`). The fresh v0513 env omitted it; the Parakeet audio
   extractor (`models/parakeet.py`) `requires_backends(librosa)` → crash at processor init. v0.5.12 env
   had 0.11.0. Installed `librosa==0.11.0` + added to setup.sh's eval-deps step.
2. **patch 063: RDNA4 `relu2` HIP fallback.** v0.5.13 imports the native `relu2` op only under `_is_cuda`
   (`sglang.jit_kernel.activation`); the `_is_hip` branch imports gelu/silu but not relu2, so
   `ReLU2.forward_cuda → relu2(x)` NameErrors on RDNA4 at cuda-graph capture (nemotron_h squared-ReLU MLP).
   Torch fallback (`max(0,x)^2`) in the HIP import branch.

After both: boots `MambaRadixCache hybrid_ssm=True`, basic+thinking **STRONG** (17×4=68 w/ reasoning).

**Verdict: parity-or-better.** v0.5.13 is at least as fast as v0.5.12 (possibly improved; conservative

**Unblock attempt (2026-06-17, CT→AWQ — partial):** converted the checkpoint CT→native-AWQ via
`convert_moe_ct_to_awq.py` (→ `GLM-4.5-Air-REAP-AWQ-native`, 42 GB, scales-check clean) to sidestep Marlin

`modules_to_not_convert` (substring `is_layer_skipped_awq`) — so the BF16-ignored Linears (layer-0 dense MLP,
shared_experts) were AWQ-loaded → `ValueError: input size not aligned` (dense MLP down_proj in=10944 → 5472
@TP=2, not %128). Fixed the tool to populate `modules_to_not_convert` from `ignore`; with that, **the native
AWQ BOOTS** (past Marlin + TP-align). **Remaining blocker: HSAIL 0x1016 on the first prefill forward** — a
deep RDNA4 quant/kernel fault (conversion qweight-packing for GLM's expert dims [intermediate 1408 = 11×128],
or moe_wna16 on GLM's 128-expert config, or a NaN cascade). The standard Step-1 isolation tool
`--enable-nan-detection` was **removed in v0.5.13** (the dropped patch 034), so isolating this needs
`HIP_LAUNCH_BLOCKING=1` + layer-bisect or re-implementing the logits-sanity check. **Deferred to a focused
HSAIL session** (task #15). Net: got from "Marlin-blocked at load" → "boots, HSAILs at prefill"; still not a
promotion blocker (pre-existing block, no regression).
