# FP8 256K inference options on 2× R9700 — 2026-07-18

This campaign establishes a post-patch-087 baseline and tests the highest-plausibility next steps for
fast 256K agentic coding. It finds and ships the largest gain since the long-context attention patches:
Laguna's native dense block-FP8 path improves decode by **+48% short and +37% at 220K**; a 58K
early-stopping run observed +45% but produced different completion lengths across the two backends.
It also fixes the benchmark instrumentation that initially hid this win, rejects invalid DCP2 topologies,
and preserves Laguna's custom tokenizer instead of applying a false-positive Mistral rewrite.

Structured runs and flags: [fp8-256k-options-r9700-2026-07-18.json](fp8-256k-options-r9700-2026-07-18.json).

## Configuration and method

- 2× Radeon AI PRO R9700 (gfx1201, 32 GiB each), TP=2.
- ROCm 7.2, PyTorch 2.11.0+rocm7.2, Triton 3.6.0.
- SGLang v0.5.15 tag source `f63458b5beaceabbd9d749b9fc956370e1b649e6`, plus patches
  001–089 (performance code through 087; correctness guards 088/089).
- `decode_ab.py`, temperature 0, up to 80 output tokens. Each run uses the API-reported
  `completion_tokens` over the first-decode-event-to-finish interval; tables report the median of five
  run-level average TPOT values.
- Laguna used its production FP8 KV metadata/scales, Triton attention, 64 KV splits, and batch-one HIP
  graph. Backend-control experiments explicitly identify `auto` or native Triton; native Triton is now
  the production default.

The first version of the harness counted nonempty SSE text events. That is not token rate when a reasoning
or tool parser buffers text, and activation FP8 changed Laguna's generated text/chunking enough to create
a false regression. The harness now counts completion tokens and has a mocked coalesced-event unit test.
All ship/reject decisions below use corrected or same-output controlled measurements.

The context sweep is increasing-depth and can reuse radix prefixes. A standalone deep run decodes
immediately after a full cold prefill, so deep auto/Triton and overlap decisions use matched five-run hot
controls after the cold prime.

## Dense Triton block-FP8 backend

Laguna previously selected an RDNA4 Torch fallback for dense block-FP8: dequantize the full weight to
BF16, then call `torch.mm`. Laguna performs 80 such dense linears per token/rank—two in layer zero and two
shared-expert projections in each of 39 sparse layers. The already-present normal Triton kernel is correct,
graph-capturable, and uses native FP8 WMMA.

| Actual input tokens | Auto tok/s | Triton tok/s | Change |
|---:|---:|---:|---:|
| 62 | 50.068 | **73.980** | **+47.8%** |
| 7,403 | 48.964 | **71.342** | **+45.7%** |
| 58,785 | 45.156 | **65.270** | **+44.5%** |
| 220,277 | 40.284 | **55.125** | **+36.8%** |

The short, 7.4K, and 220K arms had matching completion counts (80, 33, and 80 respectively). At 58.8K,
`auto` stopped after 20 tokens and Triton after 29, so +44.5% is the observed completion-token rate on
different early-stopping continuations rather than a fixed-output kernel isolation. The matching short
and deep controls establish the promotion independently of that point.

The deep pair was reverse-confirmed on fresh boots with identical token-count timing. Native FP8 also
reduced batch-one graph memory from about 0.44 to 0.29 GiB. Quality gates passed:

- comprehensive text: 35/36, including 8/8 code generation; the only miss was Python's `-7 % 3` result;
- capabilities: 2/2 basic and thinking;
- multi-turn tools: structured `read` then `todowrite`, with no tool marker leakage;
- early-needle recall: 3/3 at 58,341, 116,586, and 176,624 actual prompt tokens;
- coherent 80-token generation at 220,277 input tokens.

Laguna now defaults to `--fp8-gemm-backend triton`; the rollback is
`FP8_GEMM_BACKEND=auto ./scripts/launch.sh laguna`.

There is still tuning headroom, but the stock SGLang block-FP8 tuner is unsafe on gfx1201: it benchmarks
an unrolled-x4 kernel that serving currently does not select, proposes stages/BLOCK_M/BLOCK_K values that
production clamps or cannot run, has no numerical gate, and can issue out-of-bounds loads for Laguna's
frequent K=256 shape. A follow-up tuner must use only the normal kernel, BM16/BK128/GROUP1/stages0,
correctness and graph-replay gates, and prioritize the 39× shared shapes (N,K)=(512,2048)/(2048,256).

## Grouped decode-attention launch tuning

The current grouped GQA stage-one launch is `BLOCK_N=32`, four warps. Lower-register alternatives compile
without spills, but did not win in same-output directional tests at 220,277 tokens. These tests used the
then-default `auto` dense backend, so their absolute rates are not the current native-FP8 curve:

| Arm | VGPR / LDS | Median tok/s | Versus baseline |
|---|---:|---:|---:|
| N32 / W4 (baseline) | 120 / 4 KiB | 38.974 | — |
| N16 / W4 | 85 / 2 KiB | 35.014 | −10.2% |
| N32 / W8 | 96 / 4 KiB | 38.342 | −1.6% |

N16 doubles loop trips. N64 compiles but rises to 177 VGPR/8 KiB (W4) or 132 VGPR/8 KiB (W8), so it
was stopped before an expensive server arm. `BLOCK_H=8` is not an RDNA4 option: native WMMA requires a
16-row tile; disabling matrix acceleration spills and loses the relevant fast path. Keep N32/W4.

## Overlap scheduling

An increasing-depth overlap-on sweep on the `auto` dense backend initially appeared positive (50.907
tok/s at 62 tokens and 40.498 at 220,277). The result was a cache-state false positive. After priming the
identical 220,277-token prefix:

| Scheduler | Five runs (tok/s) | Median |
|---|---|---:|
| overlap off | 39.753 / 40.080 / 39.792 / 39.770 / 39.790 | **39.792** |
| overlap on | 39.736 / 39.267 / 39.480 / 39.800 / 39.810 | **39.736** |

That is −0.14%, effectively neutral. Laguna remains overlap-off by default. The launcher now accepts
`ENABLE_OVERLAP_SCHEDULE=1` so concurrency/shared-prefix agentic workloads can be tested without editing
the preset.

## Decode context parallelism

DCP2 was excluded on correctness grounds rather than benchmarked. SGLang's MHA DCP path all-gathers the
TP-sharded query heads across adjacent ranks while each rank retains its local K/V projection. Every rank
inside a DCP group must therefore hold the same replicated K/V heads:

```text
num_kv_head_replicas % dcp_size == 0
```

Every installed TP2 FP8 coding candidate has `num_kv_head_replicas=1`, so DCP2 would LSE-merge attention
against different KV heads. North and Laguna also lack DCP-scaled hybrid allocation, masked SWA writes,
sequence-sharded window indices, and sliding-window extend support. A fail-fast patch is preferable to a
fast but invalid number.

For reference, Qwen3-Coder-30B-A3B FP8 at valid DCP1 measured 59.677 / 53.725 / 49.988 / 42.675 /
34.915 tok/s at 20 / 29,249 / 58,479 / 116,940 / 219,244 actual input tokens. Its quadratic full-attention
prefill, not decode alone, is a major 256K agent-loop cost. Those Qwen values predate the token-count
instrumentation fix and should be refreshed before cross-backend comparison.

Patch 088 now enforces the replica condition before distributed initialization or weight loading for MHA;
MLA remains on its separate path.

## Laguna tokenizer correctness

Transformers 5.12.1 warned that Laguna should enable `fix_mistral_regex=True`, but Laguna uses Poolside's
custom 100,352-token BPE rather than the affected Mistral tokenizer. Enabling the suggestion changed IDs
in 122/223 sampled SGLang Python files, including CamelCase identifiers. Patch 089 explicitly defaults the
flag to false only for `model_type=laguna`, preserves explicit caller overrides, removes the misleading
warning, and keeps checkpoint token IDs unchanged.

## Reproducibility checks

- The completion-token accounting test passed 1/1; DCP layout/topology tests passed 8/8; the full
  tokenizer/config unit file passed 62/62.
- All 62 numeric patches applied strictly from the v0.5.15 tag source. The resulting 7,017 files/symlinks
  matched `/data/sgl-v0515` byte-for-byte, with no unexpected double-applicable patch; patch 026 remains
  the documented non-unique-anchor exception.
- A clean `./scripts/launch.sh laguna` boot selected `fp8_gemm_runner_backend='triton'`, captured the
  batch-one graph, became healthy, and completed an OpenAI chat request. The server was then stopped and
  both GPUs returned idle.

## Disposition and next experiments

1. Profile the new native-FP8 default, then resweep decode KV splits 48/64/80/96/128. Reducing dense work
   makes long-context attention a larger fraction and can shift the 64-split optimum.
2. Build a correctness-gated gfx1201 tuner for the normal dense block-FP8 kernel, starting with Laguna's
   39× shared-expert shapes. Do not use or repair only the stock unrolled selector.
3. Add a realistic agentic benchmark: cold first turn plus warm multi-turn/shared-prefix branches,
   concurrency 1–4, and p50/p95 TTFT and TPOT. The current repeated-prompt sweep is a kernel diagnostic.
4. Keep query-selected sparse decode as an opt-in candidate (previously 1.77× near 245K), but expand its
   quality gate from six instances to 30–50 coding and retrieval tasks before considering an agentic-fast
   preset.
5. Do not prioritize AITER, HiSparse, DCP2, smaller `BLOCK_H`, or a stack upgrade. Current Triton already
   emits gfx1201 FP8 WMMA; those alternatives are unsupported, incorrect here, or unproven.
