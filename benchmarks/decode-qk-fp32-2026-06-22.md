# Decode-QK FP32 accumulation — NEGATIVE (#48, 2026-06-22)

**Question:** does FP32 decode-QK accumulation improve long-KV retrieval quality? Patch 011 made the
decode **PV** product + the **extend** QK fp32-input, but left **decode-QK bf16-input** (`q_k =
q.to(K.dtype)` → `tl.dot(q_k, k)`, fp32-accum / bf16-input). (The original int4-agentic motivation —
the 3090's "triton precision" hypothesis — was already refuted separately: int4 agentic 0/6 is an int4
*weight-noise* ceiling, not attention precision; torch_native-vs-triton A/B, 2026-06-15.)

**Method:** env-gated `SGLANG_DECODE_QK_FP32` (a `QK_FP32: tl.constexpr` in `_fwd_grouped_kernel_stage1`
casting q,k → fp32 for the QK `tl.dot`). A/B on **devstral** (non-thinking, dense *softmax* attention →
the gated decode kernel runs on every layer; qwen35 was the wrong test — DeltaNet + thinking).
needle@~77K depth + steady-state decode tok/s (220-tok gen).

| arm | needle@77K | steady decode tok/s |
|---|---|---|
| baseline (bf16 QK) | PASS | 24.98 |
| fp32 QK | PASS | 11.85 |

**Result: NEGATIVE — reverted.** No quality gain (bf16 already retrieves the needle; no precision
deficit at this depth) and a **~2× decode-speed regression** (fp32 `tl.dot` has no fast WMMA path on
gfx1201). Not shipped, gate reverted. bf16 decode-QK (fp32-accum) is sufficient.
