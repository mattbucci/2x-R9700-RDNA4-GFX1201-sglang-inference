# North Mini Code + Laguna XS.2 on SGLang v0.5.15 / RDNA4

Focused single-user decode campaign on the repository's live SGLang v0.5.15 stack. The goal was to
separate launch/configuration wins from changes inside SGLang, then retain only changes that improved
this exact 2× R9700 system.

## Test system and method

- 2× AMD Radeon AI PRO R9700, gfx1201, 32 GB each; TP=2; ROCm 7.2; RCCL 2.27.7.
- Fresh conda env `sglang-triton36-v0515`; editable tree `/data/sgl-v0515`.
- SGLang reports `0.5.16.dev47+…` from the v0.5.15-based source tree.
- Triton attention and Triton MoE, FP8 checkpoint weights, FP8 KV unless an A/B says otherwise.
- CUDA/HIP graph capture at batch size 1. Decode results are median TPOT converted to tok/s. Context
  labels below identify the actual input lengths; they are not synthetic KV-pool capacities.
- Focus models: `CohereLabs/North-Mini-Code-1.0-fp8` (local copy) and
  [`poolside/Laguna-XS.2-FP8`](https://huggingface.co/poolside/Laguna-XS.2-FP8).

Raw stage values and method notes are also stored in
[`north-laguna-v0515-r9700-2026-07-12.json`](north-laguna-v0515-r9700-2026-07-12.json).

## Why the internal changes win

- The generic HIP sigmoid top-k route now uses the same fused Triton router path as CUDA instead of a
  multi-op Torch fallback. On R9700 it measured about **0.0082 ms vs 0.028–0.032 ms** (3.4–3.9×) for
  representative 128/256-expert shapes, with identical expert IDs and score error at or below 3e-8.
- Laguna's checkpoint router tensors are BF16. Keeping the gate weight in the model dtype and casting
  the logits to FP32 after the linear matches the reference order and measured about **0.020 ms vs
  0.064 ms** for FP32 gate GEMV (3.2×), with the same sampled top-k IDs.
- The old HIP communicator rule cast every attention all-reduce to FP32, doubling payload. The new
  switch preserves that defensive default for recurrent/DeltaNet users but lets Laguna use the normal
  BF16 attention collective; the gain grows with context in the measured curve.
- The v0.5.15 HIP path otherwise falls back to multi-op Torch RMSNorm because the CUDA extension is not
  present. The generalized one-launch Triton RMSNorm was about **5–5.5×** faster for the tested
  hidden-size-2048 shapes. Laguna's combined q/k RMSNorm was about **4–8×** faster across the tested
  M=1/8/64 shapes. Decode parity was exact in the M=1 cases; larger-M differences stayed at BF16-scale.
- Static-scale FP8 K/V cache writes previously executed divide, clamp, cast, and scatter as separate
  operations. The retained fused Triton store emits both K and V through one launch and was byte-identical
  to the old path at M=1/8/64/4096. Kernel speedups were **1.77× / 2.56× / 2.57× / 6.36×** respectively.
  Laguna's server reverse A/B then confirmed **+2.57% / +20.70% / +46.89%** at the three measured
  depths. North's final ON curve improved in absolute terms, but its OFF control was too unstable to
  attribute a clean percentage.

## Configuration and memory findings

- CUDA/HIP graph capture is the largest configuration-only win for Laguna: roughly 21 tok/s eager to
  30–34 tok/s under graphs across this curve.
- The model-specific MoE tuner produced about a 5.1% paired-kernel win for Laguna and 11.2% for North;
  end-to-end gains were smaller because MoE is only part of a decode step.
- Laguna `--swa-full-tokens-ratio 0.01` increased the measured token capacity from **875,509 to
  1,009,385** (+15.3%) with neutral speed and passed the eight-way serving check. It is the FP8 default.
  Forced BF16 KV required ratio 0.02, roughly halved capacity, and did not improve decode.

## Rejected experiments

- Laguna custom all-reduce plus `SGLANG_USE_1STAGE_ALLREDUCE=1`: neutral, not adopted.
- Forced BF16 KV: neutral speed and substantially lower capacity, not adopted.
- A fused Laguna attention softplus-gate Triton kernel was 2.3× faster in isolation but regressed the
  server curve to 37.085 / 36.232 / 31.423 tok/s, so it was reverted.
- A universal BF16 collective was not adopted. The narrower model-scoped switch keeps the existing FP32
  protection wherever it may still be needed.

## Correctness status

- North scored **34/36** on the final comprehensive run. One failure was an evaluator false-negative
  even though the response contained the exact speed-of-light answer; the other was the matrix test.
  Its final context curve was coherent and the tool-call probe passed.
- Laguna scored **34/36**. The speed-of-light response again contained the exact answer but was
  mis-scored, while the `-7 % 3` miss is the pre-existing model failure. Capabilities passed **2/2**,
  the tool-call probe passed, and the 220,277-input generation was coherent.
- Kernel-level router/top-k IDs matched their references, the new RMSNorm paths were checked against
  the existing PyTorch behavior on GPU, and fused K/V stores were byte-identical at every tested size.
