## The v0.5.14 theme: `get_attention_tp_rank()` ‚Üí `get_parallel().attn_tp_rank`
v0.5.14 introduced a `ParallelContext` accessor (`sglang/srt/runtime_context.py:get_parallel()`)
and migrated TP-rank/size call sites from the free functions `get_attention_tp_rank()` /
`get_attention_tp_size()` (still defined in `layers/dp_attention.py`, just no longer the idiom)
to `get_parallel().attn_tp_rank` / `.attn_tp_size`. This is what generated 3 of the 4 conflicts.
Resolution rule: **adopt `get_parallel()` at every site touched** (already imported in the
affected files; semantically identical) while preserving our RDNA4 deltas.

## 065 ‚Äî initially dropped as "upstreamed", then RESTORED (2026-06-27): the upstream is gfx95-only
v0.5.14 ships a native `layers/attention/triton_ops/verify_splitkv.py` + `verify_splitkv_fwd`, so the
rebase first archived 065 as upstreamed. **That was a mistake.** v0.5.14 **hard-gates** the native kernel
to gfx95/CDNA4 (`triton_backend.py:151` `use_verify_splitkv = is_gfx95_supported() and ‚Ä¶` ‚Äî the comment:
"unvalidated on other AMD archs, so restrict it to gfx95 and fall back to extend_attention_fwd everywhere
else"). On our **gfx1201 (RDNA4)** `is_gfx95_supported()` is **False**, so the native path **never fires** ‚Äî
archiving 065 silently dropped split-KV verify on our hardware. **Restored 065** (commit `22bf757`): the
gfx1201 `tree_verify_attention` kernel re-integrated AFTER v0.5.14's inert-on-RDNA4 native path, before the
extend fallback, gated on `SGLANG_TREE_VERIFY_SPLITKV` (set by `launch.sh --spec`). The two coexist
(complementary hardware gates). Validated: qwen36-27b FP8 EAGLE3(topk=1) spec @256K runs the kernel
correctly, no crash, 46.8 tok/s vs 14.8 no-spec at short ctx. **Lesson:** when a rebase drops a patch as
"upstreamed", check the upstream's hardware gate ‚Äî `is_gfx95_supported()`/`is_cuda()` guards can exclude RDNA4.

## ‚úÖ RESOLVED: v0.5.14 cuda-graph **warmup-capture hang** on RDNA4 = missing RCCL pre-warm
v0.5.14 restructured the cuda-graph system into `CudaGraphConfig(decode=PhaseConfig(backend='full'‚Ä¶))`
(`model_executor/cuda_graph_config.py`; backends `full|breakable|tc_piecewise|disabled`) and **deprecated
`--disable-cuda-graph`** ‚Üí `--cuda-graph-backend-{decode,prefill}=disabled`. The **`backend='full'` decode
capture wedged at warmup on RDNA4**: coder-30b booted, printed "ready", then hung before serving ‚Äî TP0 `R`/busy,
TP1 `__futex_wait`, GPU idle-low, port dead.

**Root cause (server_args.py):** v0.5.14's FULL backend captures the TP all-reduce **into** the graph; the
RCCL communicator's channel-init runs *during* `hipGraph` capture and deadlocks at TP=2. The flag that
prevents this ‚Äî `pre_warm_nccl` (one warmup `all_reduce` before capture, `model_runner.py:1244`) ‚Äî has a
docstring saying *"Default: enabled for AMD/HIP (RCCL)"* but **the code never enables it** (field default
`False`; `_handle_nccl_pre_warm()` only ever turns it *off*). So on RDNA4 it stayed off ‚Üí capture deadlocked.

**Fix (`scripts/launch.sh`, 2026-06-26):** pass **`--pre-warm-nccl`** unconditionally in the CMD builder
(HIP-only-applicable; a ~0.2s no-op at TP=1 / cuda-graph-off). Validated end-to-end through the default
`launch.sh coder-30b` path: `NCCL/RCCL warmup completed in 0.22s` ‚Üí `Capture target decode CUDA graph begin
backend=full bs=[1,2,4,8,12,16,24]` ‚Üí ready ‚Üí **serves codegen STRONG at 53.9 tok/s vs ~24 eager (~2.2√ó),
health 200 after** (no wedge). The cuda-graph decode speedup is restored. gemma4 (skip-warmup ‚Üí eager) is
unaffected and still 4/4. **Upstream PR-candidate:** default `pre_warm_nccl=True` on HIP per its own docstring
(small ROCm one-liner, same class as 031/044) ‚Äî left as launch-flag for now (lower risk than overriding a
possible explicit `--no-pre-warm-nccl`).
